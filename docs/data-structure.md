# データ構造

## 方針：数字6面 + 接着剤の関数1個

必要なのは2つだけ。立方体の複雑さはすべて幾何由来の関数（`twins` / `peers`）に閉じ込め、盤面データ自体はただの数字にする。

```ts
type FaceId = 'U' | 'D' | 'F' | 'B' | 'L' | 'R';

// ① データ = 6面ぶんの数字。これが盤面の状態のすべて。人間が読める。
type Board = {
  faces:  Record<FaceId, Uint8Array>;  // 各面81マス、0 = 空
  givens: Record<FaceId, Uint8Array>;  // 初期ヒントかどうか（同じ形の並行グリッド）
};

// ② 立方体らしさ = 幾何から決まる純粋関数（geometry.ts が構築）
//    twins(face, i): そのマスの双子（同じ辺/頂点の別面のマス）0〜2個
//    peers(face, i): 同じ数字になってはいけない相手すべて
```

- `i` は面内のマス index（0..80、`r*9 + c`）
- 保存・共有・デバッグは「面ごと81文字 × 6」にシリアライズできる。人間可読で、幾何の実装を変えても壊れない

## 入力時の同期

辺のマスに数字を入れたら、その双子にも同じ数字を書く（1行）：

```ts
function setCell(board, face, i, value) {
  board.faces[face][i] = value;
  for (const [tf, ti] of twins(face, i)) board.faces[tf][ti] = value;
}
```

これで「辺の2マス（頂点なら3マス）が食い違う」状態が構造的に起きない。

## varId / union-find について

ソルバを速く回すために、内部では共有マスを1つの変数に畳んだ 386個の配列を使うことがある（[geometry.md](geometry.md) の386変数）。ただしこれは**ソルバの実装詳細**であり、`Board` の表には出さない。アプリが扱うデータ構造は「6面の数字 + `twins()` / `peers()`」だけ。

## なぜ面ベースを正本にするか

- **可読性**：386個の数値配列は人間もデバッグも読めない。6面×81文字なら目で読める
- **幾何非依存**：辺対応のコードを直しても、保存済みの盤面が壊れない
- **既存資産との互換**：面ごと81文字は通常の数独の入出力とそのまま繋がる
