# 問題生成ロジック

Sudocube の問題生成は、通常の数独とまったく同じ骨格。違うのは制約（`peers()`）だけで、アルゴリズムは変えない。

## 全体パイプライン

1. **完成盤面を生成**する（すべての制約を満たす、矛盾のない solved 盤面）
2. 完成盤面から**マスを削る（掘る）**
3. 1マス削るごとに**ソルバで「解が唯一か」を判定**する
4. ユニーク解が保たれる限り掘り続ける。掘れなくなったら問題完成

ポイント：**ソルバは「答えを出す道具」ではなく「解が1個だけか数える道具」として使う**。解が2個見つかった時点で、その掘り方はボツと判定する。

## ソルバ

- 手法：**DFSバックトラック + MRV**（Minimum Remaining Values＝候補が最も少ないマスを先に埋めるヒューリスティック）
- 制約：あるマスに置ける数字は、`peers()` が返す相手の既出値を除いたもの
- 数独は深さが固定（全マス）なので DFS。BFS はメモリが爆発するので使わない

### 解の個数カウント

```ts
// 解を cap 個見つけたら即打ち切る。一意性判定には cap = 2 を使う。
countSolutions(board, cap): number
```

- `countSolutions(board, 2) === 1` なら唯一解（＝問題として成立）
- `=== 0` なら解なし、`>= 2` なら複数解（掘りすぎ）

全解を数え上げる必要はない。2個目が見つかった瞬間に止めるのが速い。

## 完成盤面の生成

- **seed は使わない**。通常の数独では「対角3ブロックをランダムに埋める」高速化テクを使うが、Sudocube では各面の seed が辺で繋がった隣の面と衝突しうるため、最初から全マスを DFS で埋める
- 検証実験では、seedなしDFSで **8ミリ秒・416分岐**で完成盤面が1個得られた（過剰拘束ではないことを確認済み）

## 掘り方

- 全マスの index をシャッフルし、順に「試しに空にして `countSolutions == 1` を保てるか」を確認する
- 1マスがダメでも終了せず、別のマスを試す。**抜けるマスを尽くす**まで回すと、より難しい問題になる
- 抜く順で難易度が変わる（難易度調整のレバー）

## 未検証：難易度は成立するか（次の検証ポイント）

制約が密なため、**2つの力が逆向きに働く**：

- 制約がきつい → 抜けるマスが少ない → ヒントが多い → 簡単？
- 面が辺で繋がっている → 1つのヒントの情報が隣の面まで波及する → 少ないヒントでも一意に決まる → もっと抜ける？

どちらが勝つかは、実際に掘って「何マスまで抜けたか（ヒント数）」を測るまで分からない。生成器を実装したら、複数回生成してヒント数の min / max / 平均を取り、難易度が成立するか（スカスカでも解ける歯応えのある問題ができるか）を確認する。これが生成ロジック実装後の最初の検証課題。
